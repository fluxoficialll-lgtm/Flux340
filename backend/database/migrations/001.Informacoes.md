# Documentação da Camada de Migrações (Migrations)

O diretório `/backend/database/migrations` é um dos mais críticos do projeto. Ele contém os scripts SQL que definem e evoluem a estrutura (o "schema") do banco de dados PostgreSQL.

## Estrutura e Funcionamento

Os arquivos são numerados em ordem sequencial para garantir que sejam executados na ordem correta. Cada arquivo representa uma alteração incremental no banco de dados.

- **Nomenclatura:** `NN_Descricao.da.Tabela.sql`
  - O prefixo numérico (`01`, `02`, etc.) estabelece a ordem de execução.
  - A descrição ajuda a identificar rapidamente qual entidade do banco de dados o script manipula.

### Principais Responsabilidades:

- **Definição de Tabelas:** A responsabilidade primária destes arquivos é criar as tabelas do banco de dados usando o comando `CREATE TABLE`.
- **Estrutura de Dados:** Eles definem todas as colunas de uma tabela, seus tipos de dados (ex: `VARCHAR`, `INT`, `TIMESTAMPTZ`, `SERIAL PRIMARY KEY`), restrições (`NOT NULL`, `UNIQUE`) e valores padrão (`DEFAULT`).
- **Relacionamentos:** Definem as chaves estrangeiras (`FOREIGN KEY`) que estabelecem os relacionamentos entre as diferentes tabelas, garantindo a integridade referencial dos dados.
- **Versionamento do Schema:** O conjunto de migrações funciona como um sistema de controle de versão para o banco de dados. Qualquer nova tabela, coluna ou alteração estrutural deve ser feita criando um novo arquivo de migração numerado.

## Processo de Execução

Quando o backend é iniciado, um script (geralmente localizado no arquivo principal do servidor ou em um arquivo de inicialização do banco de dados) lê todos os arquivos deste diretório em ordem numérica e os executa um a um contra o banco de dados.

Este processo garante que a estrutura do banco de dados esteja sempre consistente com o que o código da aplicação espera. Se uma nova migração é adicionada, ela será executada automaticamente na próxima inicialização, atualizando o banco de dados sem a necessidade de intervenção manual.

**Atenção:** Uma vez que uma migração é aplicada a um ambiente (especialmente produção), ela nunca deve ser alterada. Se uma mudança for necessária, deve-se criar uma *nova* migração (ex: `17_Altera.Tabela.Existente.sql`) que aplique as modificações desejadas (`ALTER TABLE`).
